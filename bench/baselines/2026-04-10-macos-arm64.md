## Cross-tool wall-clock (`bench/run.sh cold-install` / `warm-install` / `up-to-date`)

All numbers are wall-clock milliseconds, median of 3 runs. All tools run with `--ignore-scripts` (or LPM's two-phase model that doesn't run scripts at install time anyway). Lower is better.

| Scenario | npm | pnpm | bun | **lpm** |
|---|---:|---:|---:|---:|
| **Cold install** (no cache, no lockfile) | 2436 | 2052 | 1199 | **2567** |
| **Warm install** (lockfile + store, fresh `node_modules`) | 932 | 921 | 318 | **402** |
| **Up-to-date** (everything in place, no-op fast path) | 433 | 235 | 9 | **52** |

### Reading the table

- **Cold install**: LPM is the slowest of the four. Slower than npm. **This is a real issue worth fixing in a future phase** — see analysis below.
- **Warm install**: LPM beats npm and pnpm comfortably (~2x faster) but bun is still ~25% faster.
- **Up-to-date**: LPM is dramatically faster than npm/pnpm (~5-8x), but bun is ~6x faster again. Most of LPM's 52ms is process startup, not install work — see per-stage data below.

## LPM per-stage (`bench/run.sh lpm-stages`)

Stage breakdown via `lpm install --json` parsing. LPM-only because no other package manager exposes a stable structured stage shape. All numbers in milliseconds, median of 3 runs.

| Scenario | resolve | fetch | link | **total** |
|---|---:|---:|---:|---:|
| **Cold** (`LPM_BENCH_ALLOW_WIPE=1`) | 1031 | 372 | 54 | **1449** |
| **Warm** | 0 | 0 | 75 | **79** |
| **Up-to-date** | 0 | 0 | 0 | **0** (sub-millisecond) |

### Reading the table

- **Cold install resolve dominates** (1031 / 1449 ≈ 71% of total). The resolver is the cold-path bottleneck. This is the obvious target for cold-install optimization in a future phase.
- **Cold per-stage total (1449ms) is ~1.1s less than wall-clock total (2567ms).** The gap is bench-harness overhead: setup, binary startup, JSON serialization, shell loop, median computation. The per-stage number is closer to "what the install pipeline actually does." Both are valid measurements for different purposes.
- **Warm install link is 75ms** of 79ms total — nearly all the warm time is linking. Resolve and fetch are skipped because the lockfile is the source of truth. Nothing further to optimize on the warm path until the linker becomes the bottleneck of something larger.
- **Up-to-date is sub-millisecond inside the install pipeline.** The 52ms wall-clock is essentially binary startup (cargo --release LTO is good but not bun-good) + arg parsing + the up-to-date check itself. The pipeline is doing the right thing.

## Observations and follow-on items

### LPM is slow on cold install vs npm/pnpm/bun

LPM's cold install (2567ms wall-clock) is the slowest of the four package managers tested. Per-stage data shows the resolver is the dominant cost (1031ms). Possible explanations to investigate:

1. **Resolver round-trips**: even with the deep batch metadata prefetch, resolution is still doing per-package metadata work that other PMs may parallelize differently. Worth profiling with `tracing` enabled.
2. **First-install cost amortization**: this benchmark wipes the cache before every run. Real-world cold installs are rare. Bun and pnpm both maintain global stores that survive across projects, and their "cold" benchmark is closer to "fresh project, warm store." LPM's `--allow-new` flag plus the wipe means we're measuring true first-ever-install on this machine, which is a worst case.
3. **Network latency variance**: a single run is sensitive to upstream registry latency. The median of 3 reduces this but doesn't eliminate it.
4. **TLS handshake / connection pool**: if LPM creates a fresh client per request internally, that could add hundreds of milliseconds vs reused connections.

**Action**: do not panic. Cold install is the rarest scenario in real workflows. The warm and up-to-date numbers are healthy. Tracking this regression is exactly why we're recording the baseline. A future phase (likely after Phase 1's filter engine ships) should profile and optimize the cold resolve path.

### Phase 34 correction: the cold-path bottleneck is mostly network/server time, not Rust compute

The Phase 34 profiling pass and the 2026-04-13 validation rerun materially
change the interpretation:

- The **command-only** improvement (`48ms` → `8ms`) proves the fast-lane/startup
  work landed as intended.
- The **engine cold** comparison (`1481ms` before vs `1470ms` after) shows no
  reproducible regression in the install pipeline itself.
- The dedicated cold profiling in [bench/profiling/cold-resolve-2026-04-12.md](../profiling/cold-resolve-2026-04-12.md)
  shows `batch_metadata_deep` network/server wait at roughly **2066ms of 2322ms
  cold resolve** (~89%).

That means the next serious cold-install win is unlikely to come from shaving a
few milliseconds off local Rust code like `to_pubgrub_ranges` or startup logic.
The highest-leverage targets are server/protocol level:

1. **Reduce `batch_metadata_deep` TTFB**: instrument the server route end-to-end
	and find out whether the time is spent in auth, dependency expansion,
	upstream npm metadata fetches, serialization, or response buffering.
2. **Compare deep-batch vs shallower parallel fetch**: a single large request
	may be over-aggregating and increasing time-to-first-byte. The data should
	decide whether the deep prefetch is actually winning.
3. **Minimize payload and duplicate work**: if the server sends much more
	metadata than the resolver needs for first-pass resolution, trim it.
4. **Only then tune client compute**: client-side parse/cache-write work is on
	the order of tens of milliseconds, not hundreds.

### LPM up-to-date startup overhead is real

52ms for "do nothing" is dominated by binary startup + arg parsing. Bun's 9ms is the gold standard. Two paths to closing the gap:

1. Smaller binary (less LTO work, fewer dependencies pulled in for the startup path)
2. Earlier exit before tokio runtime spin-up when the up-to-date check can be done synchronously

This is a Phase 9 (`--report-json`) concern at the earliest. Document, don't fix yet.

## Measurement Classes (Phase 34.3)

Every benchmark number belongs to one of three classes. The class determines what the number includes and how to interpret regressions.

| Class | What's measured | Includes | Example benchmarks |
|---|---|---|---|
| **wall-clock** | Full shell command with orchestration | `rm -rf`, cache wipes, process startup, shell loop | `cold-install`, `warm-install`, `up-to-date` |
| **command-only** | Binary invocation on prepared fixture | Process startup, arg parsing, install logic | `command-only` (Phase 34.3) |
| **engine** | JSON per-stage timing from `--json` | Resolve, fetch, link — no startup or I/O overhead | `lpm-stages` |

When investigating a regression:
- **wall-clock regressed but engine didn't** → harness overhead or startup regression
- **engine regressed but wall-clock didn't** → engine slowed but was hidden by faster startup
- **command-only regressed but wall-clock didn't** → binary startup cost, not install logic
- **all three regressed** → real install-path regression

### Phase 32 guardrail #3 anchor numbers

Phase 32 hard guardrail #3 says install-path features must not regress these numbers. The numbers to defend, on macOS arm64:

| Metric | Class | Budget | Baseline |
|---|---|---:|---:|
| Cold total wall-clock | wall-clock | ≤ 2700 ms | 2567 ms |
| Cold resolve | engine | ≤ 1100 ms | 1031 ms |
| Cold fetch | engine | ≤ 410 ms | 372 ms |
| Cold link | engine | ≤ 60 ms | 54 ms |
| Warm total wall-clock | wall-clock | ≤ 420 ms | 402 ms |
| Warm link (per-stage) | engine | ≤ 85 ms | 75 ms |
| Up-to-date wall-clock | wall-clock | ≤ 60 ms | 52 ms |
| Up-to-date command-only | command-only | ≤ 30 ms | ~23 ms (Phase 34.1) |
| Up-to-date per-stage total | engine | ≤ 2 ms | 0 ms |

Any PR that pushes a number above its budget needs explicit, justified discussion in the PR description.

## Limitations of this baseline

- **macOS arm64 only.** Linux x64 baseline is required before Phase 32 guardrail #3 can be called complete. See [Phase 0 status doc](../../../../a-package-manager/DOCS/new-features/37-rust-client-RUNNER-VISION-phase32-phase0-status.md) for the sequenced plan.
- **No isolated network.** Numbers can fluctuate by upstream registry conditions. Re-running on a different day will produce different numbers — variations under ~10% should be ignored.
- **Single fixture (51 packages).** Larger workspaces (500+, 5000+) may exhibit very different bottlenecks. Phase 1's filter engine work should add larger fixtures.
- **`--ignore-scripts` for npm/pnpm/bun, two-phase install for LPM.** This is the right comparison but it means we're not measuring lifecycle script overhead anywhere. That's the `script-overhead` benchmark's job.
- **Median of only 3 runs.** Adequate for tracking regressions, not enough for publication-grade numbers.

## How to reproduce

```bash
# Build the release binary
cd /path/to/lpm/rust-client
cargo build --release --package lpm-cli

# Cross-tool wall-clock (non-destructive except for cold)
bash bench/run.sh cold-install
bash bench/run.sh warm-install
bash bench/run.sh up-to-date

# LPM per-stage breakdown (warm + up-to-date are non-destructive)
bash bench/run.sh lpm-stages

# LPM per-stage including the destructive cold run
LPM_BENCH_ALLOW_WIPE=1 bash bench/run.sh lpm-stages
```

The `LPM_BENCH_ALLOW_WIPE=1` env var is required to run the cold per-stage benchmark because it wipes `~/.lpm/cache` and `~/.lpm/store`.
